## Architecture

The app is a single-page map viewer. All UI state lives in `App.tsx` and is passed down as props — there is no global state manager.

**Data flow:**
1. `DataContext` fetches `public/cities.json` on mount, builds a `cityIndex` (Record<city_code, City>), and exposes `{ cities, years, computed }` to the tree.
2. `computed` contains pre-calculated scales (`scales`, `rentScales`, `yieldScales`, `changeScales`) generated by `process.py` — these are percentile ranges used for color mapping.
3. `MapView` holds a MapLibre map instance. On every state change (mode/filter/year/showChange), it calls `applyAllFeatureStates` which iterates all ~35 000 cities and calls `map.setFeatureState` to color each polygon via MapLibre feature state.
4. Tooltips are built inside the `mousemove` handler using a `stateRef` pattern — a ref kept in sync with current props on every render — to avoid stale closures.

**Key files:**
- `src/config.ts` — `FILTER_FIELDS` (maps FilterType → YearData field names), `getModeConfig` (per-mode display config: labels, formatters, tooltip HTML), `changeScaleKey` (canonical key for change scale lookups), `getScaleForMode`.
- `src/data.ts` — pure functions: `getValue`, `getChange`, `getValidatedChange` (filters by MIN_SAMPLES_PER_YEAR=15), `computeFeatureState`, `getTooltipData`, `applyAllFeatureStates`, `valueToColor`/`changeToColor`.
- `src/types.ts` — `City`, `YearData`, `ComputedData`, `TooltipData`, `FilterType`, `ModeType`.

**Map tiles:** Vector tiles are served as a `.pmtiles` file via the `pmtiles` protocol registered at module level in `MapView.tsx`. Cities are identified by `code` property, promoted as feature ID (`promoteId`).

**i18n:** French (default) and English. Translations in `src/locales/`. The `t()` function is passed into `getModeConfig` so formatters produce translated strings.

**Data generation:** `process.py` uses DuckDB to process DVF (real estate transactions) + rent CSV files and outputs `public/cities.json` and `public/cities.pmtiles`. Requires `tippecanoe` and a Python venv with `duckdb`.

**Tests:** Vitest + jsdom + Testing Library. Test fixtures are in `src/test/fixtures.ts` (TEST_CITIES, TEST_YEARS, TEST_COMPUTED with pre-computed scales). `renderWith` in `src/test/renderWith.tsx` wraps components with i18n + DataContext providers.

**Linter:** Biome (not ESLint/Prettier). `useExhaustiveDependencies` is disabled. `noDangerouslySetInnerHtml` is disabled (used intentionally for SVG arrow injection in map popups and a few React components).

**Deploy:** GitHub Actions pushes `dist/` to GitHub Pages on every push to `main`. Base URL is `/french-real-estate-viewer/`.

## Workflow Orchestration

### 1. Plan Node Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately - don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop
- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

### 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes - don't over-engineer
- Challenge your own work before presenting it

### 6. Autonomous Bug Fixing
- When given a bug report: just fix it. Don't ask for hand-holding
- Point at logs, errors, failing tests - then resolve them
- Zero context switching required from the user
- Go fix failing CI tests without being told how

## Task Management

1. **Plan First**: Write plan to `tasks/todo.md` with checkable items
2. **Verify Plan**: Check in before starting implementation
3. **Track Progress**: Mark items complete as you go
4. **Explain Changes**: High-level summary at each step
5. **Document Results**: Add review section to `tasks/todo.md`
6. **Capture Lessons**: Update `tasks/lessons.md` after corrections

## Core Principles

- **Simplicity First**: Make every change as simple as possible. Impact minimal code.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.
